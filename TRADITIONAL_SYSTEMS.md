# Традиционные системы ТОиР / FSM (original)

**Дата:** 2026-07-11  
**Назначение:** baseline «как было» до современного FSM UI — для сравнения с Fixaverse и позиционирования vs HubEx.  
**Связанные:** [FSM_PROCESS.md](FSM_PROCESS.md), [../masterdoc/COMPETITOR_BATTLECARDS.md](../masterdoc/COMPETITOR_BATTLECARDS.md)

---

## Суть

В классическом CMMS / FSM / «1С + Excel + телефон» процесс **размазан по модулям и каналам**, а не один поток с доской. Традиционная автоматизация = **учёт факта**; современная = **оркестрация** (кто что делает сейчас).

HubEx у реальных клиентов (см. [Максим/README.md](../Максим/README.md)) — уже **средний класс** FSM, но типичная боль та же: склад и ППР **хуже связаны**, чем в демо.

---

## Как устроено по шагам

### 1. Заявка

Клиент звонит / пишет. Диспетчер заводит карточку вручную. ППР часто **отдельно**: график в Excel, ремонт — в другой очереди.

### 2. Диспетчер

Не канбан статусов, а список / таблица + телефон инженерам. Назначение = звонок или смена поля «исполнитель». Статусы обновляют **поздно** или не обновляют.

### 3. Инженер

Маршрут: бумага, WhatsApp, «мои заявки». На объекте при нехватке запчастей — **звонок диспетчеру или складу**. В системе наряд может оставаться «в работе», склад о нём не знает.

### 4. Склад

Отдельный контур: заявка на МТР в другом экране / 1С. Кладовщик видит очередь склада, **не** выезды. Связь наряд ↔ выдача — номер вручную или «если вспомнили».

Нет автоматического «Ждёт запчасть» на доске — диспетчер узнаёт из чата.

### 5. Закрытие

Акт / фото — отдельно. Наряд закрывают без проверки склада. Списание материалов догоняют позже.

---

## Схема потоков

```text
Клиент ──телефон──► Диспетчер ──звонок/чат──► Инженер
                         │                       │
                    (карточка WO)           (работы в поле)
                         │                       │
                         └── чат/бумага ──► Кладовщик ──► выдача
                                              │
                                         (учёт в 1С/складе)
```

Три роли — **три мира**, склеенные звонками.

---

## Сравнение с современным FSM

| | Традиционно (original) | Современный FSM UI |
|--|------------------------|-------------------|
| Единица работы | Заявка + побочные артефакты | Один work order со связями |
| Назначение | Вручную / телефон | Доска + календарь (+ авто-scheduling) |
| Склад | Отдельная заявка | Дочерняя к наряду |
| Блокер по запчасти | В чате | Колонка / флаг на наряде |
| Инженер | Чаты | Кнопки + «материалы» из карточки |
| ППР | Отдельный график | План → тот же наряд |

---

## Почему так держалось

1. Склад и выезд внедряли разными отделами / вендорами.  
2. Правило «нельзя закрыть без выдачи» ломало грязные процессы — не включали.  
3. Телефон дешевле внедрения связки.  
4. В РФ часто: **CRM/заявки + 1С-склад + мессенджер** — три системы навсегда.

---

## Что это значит для Fixaverse

**Дифференциатор:** не ещё один журнал, а **жёсткая связка наряд ↔ склад ↔ статусы** + copilot поверх (завод / сложное оборудование).

**Продажа vs HubEx:** не «у вас Excel», а «склад и ППР не дотягивают в текущем FSM» + TCO и миграция (ориентир HubEx ~80 тыс. ₽/мес на 40 лиц. — см. Максим).

**Риск:** если повторить традиционную схему (три несвязанных экрана), ИИ только ускорит хаос.
