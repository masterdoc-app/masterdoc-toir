## Вывод из лидов

Люди ищут «систему ТОиР», но часто имеют в виду **выездной сервис** (ППР, диспетчер, склад), а не заводской CMMS с простоями линий. Эталонный кейс: сервисная компания с сотнями договоров, ~40 лицензий FSM, роли инженер / диспетчер / кладовщик.

Два сегмента Fixaverse:

| Сегмент | Пример | Ядро UI |
|---------|--------|---------|
| **Завод / цех** | fixaverse.ru SEO, copilot у станка | Наряд + память машины + ИИ |
| **Выездной FSM** | АППЕС, HubEx-миграция | Наряд на выезд + доска + склад |

Ниже — **общая модель FSM**, применима к обоим; завод добавляет copilot и радар поломок поверх.

## Роли (MVP)

| Роль | Задача | Интерфейс |
|------|--------|-----------|
| **Заявитель** | Объект, симптом, срочность | Веб / звонок → карточка |
| **Диспетчер** | Назначить, SLA, координация | Доска + календарь + карта |
| **Инженер** | Выезд, работа, акт, материалы | Мобилка |
| **Кладовщик** | Сборка / выдача по заявкам | Очередь склада |
| **Руководитель** | Просрочки, загрузка, экономика | Сводка (не полный канбан) |

Позже: бригадир, офис договоров, бухгалтер.

## Главная сущность — наряд на выезд

Не «задача», а **work order / наряд**:

- объект, адрес, контакт
- тип: **аварийная / ремонтная / ППР**
- оборудование (если известно)
- приоритет, дедлайн, исполнитель
- дочерние **заявки на склад** (0…N)
- статус жизненного цикла

Склад — **дочерние заявки**, привязанные к наряду, не отдельный мир.

---

## Доска диспетчера

Колонки по **этапу сервиса**:

```text
Новые → Назначено → В пути → На объекте → Ждёт запчасть → На проверке → Закрыто
```

| Колонка | Кто двигает | Смысл |
|---------|-------------|-------|
| Новые | система / диспетчер | заявка без исполнителя |
| Назначено | диспетчер | инженер назначен, выезд не начат |
| В пути | инженер | выехал |
| На объекте | инженер | работы идут |
| Ждёт запчасть | инженер / диспетчер | блокер по складу |
| На проверке | диспетчер / руководитель | акт, фото |
| Закрыто | диспетчер | готово |

ППР — отдельный фильтр «плановые на неделю», те же колонки после активации.

Карточка: объект, тип, приоритет, инженер, SLA, бейдж склада (ждёт / собрано / выдано).

---

## Очередь склада (отдельный вид)

```text
Запрошено → В сборке → Готово к выдаче → Выдано → Отклонено (нет на складе)
```

В карточке: номенклатура, кол-во, **ссылка на наряд**, кто запросил, срочность.

---

## Уведомления по ролям

**Диспетчер:** новые + просрочки SLA + «склад заблокировал наряд». Push только на аварийные, срыв SLA, отказ склада.

**Инженер:** «Мои наряды» (сегодня / назначенные). Push при назначении и смене статуса склада по его наряду.

**Кладовщик:** только складские заявки; сортировка: аварийные → дата выезда → обычные.

## Кто создаёт заявку на склад

| Ситуация | Кто |
|----------|-----|
| ППР с типовым расходником | **Система** (шаблон) → инженер подтверждает |
| Симптом известен до выезда | **Диспетчер** или правило |
| Неожиданная поломка | **Инженер** |
| Типовая авария с BOM | **Система** предлагает → инженер «заказать» |

Принцип: **система предлагает и связывает, человек утверждает.**

MVP: кнопка «Материалы» на наряде; шаблон → черновик; инженер подтверждает / дополняет → «запрошено».

---

## Правила (иначе доска развалится)

1. Нельзя закрыть наряд при открытых складских строках (кроме явной отмены).
2. «Ждёт запчасть» — только при открытой складской заявке.
3. Диспетчер назначает инженера (аварийки не «берут с улицы» без правила).
4. Одна доска статусов; различия — метки (авария / ремонт / ППР).

Заявитель **не** пишет напрямую на склад — только через наряд.

## Автоматизация (ориентир западных FSM)

| Этап | UI-автоматизация |
|------|------------------|
| Приём заявки | 70–90% |
| Назначение | 60–85% рутины |
| Статусы выезда | 80–95% |
| Запрос запчастей | 50–70% |
| Сборка на складе | 40–60% |
| Блокер «нет на складе» | 70–90% при настройке |
| Диагностика | 10–30% (+ ИИ) |

Склад — слабое звено даже у зрелых FSM без резервов в той же системе.

## Сдвиг с ИИ (2025–26)

| Было | Становится |
|------|------------|
| Человек гоняет статусы | Система предлагает шаг |
| Диспетчер = оператор таблицы | Диспетчер = исключения |
| Поиск в PDF | Copilot: симптом → процедура |
| Заявка на склад с нуля | Черновик BOM из типа работ |
| Отчёт руками | Черновик из голоса / фото / чеклиста |

ИИ работает только если базовый процесс **наряд ↔ склад ↔ роли** уже связан.

## Won't-have в MVP FSM

- Маршрутизация по пробкам
- Полный WMS
- 1С-обмен
- Predictive без telematics / истории

Сначала: **заявка → назначение → склад → закрытие** каждый день.
